# 0062 — Navigation tree pattern

## Context

The product-specs and designs folders today are flat lists of artifacts
plus a registry that lists every entry side-by-side. Workers (PM,
Architect, Team Manager, Reviewer) and humans landing on the repo have
no curated entry point. To understand the system they either:

1. Read the registry and guess relevance from titles, or
2. Read every active spec, or
3. Walk the cross-link graph (`related_specs`, `served_by_designs`)
   spec-by-spec, hoping the author maintained the edges they need.

At 21 active product specs + 17 active designs, (1) works. Current PM
runs typically read 3-5 grounding files to draft a spec — the right
ones, picked from titles. At 100+ specs the title-only signal will
not be enough; the worker will either over-fetch (cost regression) or
miss relevant context (quality regression).

The seeds of a tree are already here:

- `designs/active/system-overview.md` is *de facto* an engineering
  root entry point.
- `designs/active/worker-roles.md` is *de facto* a category-level
  doc that points at architect-worker, pm-worker, etc.

But the role of "entry point" is implicit, the convention isn't
documented, and the product-specs side has no equivalent at all.

## Decision

**Adopt a hierarchical navigation pattern: root → category → leaf,
with leaves promoting to entry points when their content gets large
enough.** One unified, generated index covers product specs and
designs together (ADR 0029); the per-folder hand-curated INDEX.md
files this design originally proposed are retired in favour of a
single generated `system/INDEX.md`.

### The three levels

```
system/INDEX.md                            root entry point (generated)
  └─ {type}/active/<category>.md           category entry point (a regular spec/design with children)
       └─ {type}/active/<leaf>.md          leaf (regular spec/design)
```

- **Root** (`system/INDEX.md`): one file for the whole system.
  Generated by `scripts/render_index.py` from `parent:` chains
  across both `system/product-specs/registry.yaml` and
  `system/designs/registry.yaml`. Lists every category with its
  spec children and design children together; each leaf rendered
  with its `summary:` field. Lifecycle is the same as
  `REGISTRY.md` — YAML is the source of truth, MD is a generated
  view, drift is caught by the validator.
- **Category** (`<category>.md`): a regular spec/design that happens
  to have many children. Same frontmatter shape as any other; its
  body opens with a `## Components` section listing children. Picked
  by the curator at category-creation time. A category may exist as
  a spec, a design, or both — the index renders what exists.
- **Leaf**: every other spec/design. Names its parent in frontmatter
  and carries a `summary:` field for the index.

A leaf with internal sub-parts can graduate to an entry point by
acquiring children of its own — the schema is recursive.

### Schema additions

Two optional frontmatter fields on specs and designs:

```yaml
parent: <category-id>     # the spec/design id of this artifact's parent.
                          # Omitted on root-level entries (categories that
                          # report directly into system/INDEX.md).

summary: short one-line description (≤ 140 chars) used by the index
                          # renderer in lieu of the title. Omit and the
                          # renderer falls back to the title verbatim.
```

`parent` references an id that must exist in either
`product-specs/registry.yaml` or `designs/registry.yaml` — the
shared navigation tree spans both folders.

The reverse relationship (`children:`) is *derived* at render time
from `parent:` rather than hand-maintained, to avoid the
double-bookkeeping that bites `related_specs` today.

### INDEX.md convention

`system/INDEX.md` is generated; hand edits are lost on the next
`scripts/render_index.py` run. The validator's drift check (the same
shape applied to REGISTRY.md) keeps the file in sync with its
sources.

Generated body shape:

```markdown
# Coder system — index

> Generated from registry.yaml + parent: chains. Do not hand-edit.

[Preamble pulled from system-overview's summary.]

## <Category 1 — short name>
[Category description, pulled from the category artifact's summary.]

- Spec: [<category-id>](./product-specs/active/<category-id>.md)
- Design: [<category-id>](./designs/active/<category-id>.md)

  - [<leaf-id>](./<folder>/active/<leaf-id>.md) (spec) — summary
  - [<leaf-id>](./<folder>/active/<leaf-id>.md) (design) — summary
  - …

## <Category 2>
...
```

A category may have a spec, a design, both, or neither (rare).
Children are mixed in a single flat list per category, tagged with
their kind. WIPs are excluded — the index renders the active surface
only. WIPs live in `registry.yaml` and `ROADMAP.md`.

### Worker prompt impact

PM, Architect, Reviewer, and Team Manager task contracts read
`system/INDEX.md` first regardless of role. Cross-cutting decisions
(a Reviewer noticing a missing spec while gating a design) require
no second fetch. `Next free spec ID` (added in design 0057's
run-context block) keeps working as before.

## Categories today

The unified index reads from the `parent:` chains; the live category
set is whatever the registries currently express. The five top-level
categories shared by both surfaces today:

- **pipeline-operations** — how tasks flow, how stalls recover, how
  state surfaces.
- **worker-roles** — the role-typed worker subprocesses, their input
  contract, and how they compose into a pipeline.
- **tenancy-and-access** — multi-tenancy enforcement and actor
  attribution.
- **knowledge-and-admin** — how each project's knowledge is read,
  written, kept current, and surfaced.
- **delivery-and-infra** — how code reaches production and how the
  system itself stays maintainable.

Engineering-only sub-trees that don't have a paired product spec
(e.g. `knowledge-stack`, `coder-core-modular-monolith`) hang off the
matching product category as designs without spec siblings — the
renderer handles asymmetric children naturally.

Re-parenting a leaf is a 1-line frontmatter change plus a registry
update; the next `render_index.py` run reflects the new tree.

## Out of scope

- **ADRs** stay flat. They're append-only chronological decision records;
  thematic grouping is a future view-only concern. INDEX.md not added.
- **Bidirectional `children:` field**. Derived from `parent:` at view
  time only. Prevents the maintenance drift we already see with
  `related_specs`.
- **Multi-parent (DAG) navigation**. A leaf has *one* parent. Cross-links
  via `related_specs` cover the secondary-relationship case.
- **Deep trees**. Two levels is enough for the current scale. The
  schema doesn't forbid deeper trees, but we won't author them yet.
- **`type: category` artifact type**. Category nodes are regular
  specs/designs with children — no new type, no schema split.

## Risks

1. **Category drift**: clusters that look right today may not in 6
   months. Mitigated by: categories are regular specs/designs,
   re-parenting a leaf is a 1-line frontmatter change + registry
   update.
2. **Worker prompt cost regression**: prepending INDEX.md content
   could add a few KB to every assembled prompt. With prompt
   caching that cost is paid once, then cache-hot.
3. **Generator coupling**: a bug in `render_index.py` regresses every
   worker's grounding at once. Mitigated by the same drift-check
   pattern that guards REGISTRY.md, plus the fact that the renderer
   reads only frontmatter and produces deterministic output.

## Evolution

- **2026-05-02 — original shape (per-folder hand-curated INDEX.md).**
  Two `INDEX.md` files (one per artifact-type folder), authored by
  hand. Worker contracts injected the per-role index. The taxonomy
  was duplicated across the two files and went stale within days
  whenever an artifact moved between `wip/` and `active/`.
- **2026-05-07 — unified, generated `system/INDEX.md`
  (ADR [0029](../../../adrs/0029-unified-generated-knowledge-index.md)).**
  One file at the system level, generated by
  `scripts/render_index.py` from `parent:` chains and a new
  `summary:` field on each artifact. Validator drift-checks it the
  same way it drift-checks REGISTRY.md. Workers fetch
  `system/INDEX.md` regardless of role.
